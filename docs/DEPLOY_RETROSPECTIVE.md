# 배포 회고 및 파라미터 운영 노트

- **대상**: v0.1.0 라인 배포 마무리 (Oracle Cloud Always Free VM · 단일 컨테이너)
- **작성일**: 2026-07-13
- **범위**: LLM 대화 고도화 이후 발견한 문제 진단 → 수정 내역, 그리고 앞으로 AI 파라미터를 어떻게 제어할지에 대한 정리.

---

## 1. 이번 라운드에서 개선한 내용

### 1-1. LLM 대화 고도화 (`/api/ai/chat`)
- 초안 생성 이후 채팅이 **모든 입력을 수정 요청으로 간주**해 무조건 재생성하고, 고정 문구("요청하신 사항을 계획에 반영했습니다")로 답하던 방식을 제거.
- 신규 `/api/ai/chat`가 **현재 계획 + 최근 대화 이력 + 유저 메시지**를 LLM에 넘겨 의도를 판단:
  **계획 수정 / 질문·잡담 / 불명확** → 각각 수정+설명 / 자연어 답변 / 되묻기.
- 응답 계약: `{"reply", "planUpdated", "tasks?"}`. 실제로 계획이 바뀐 경우에만 체크리스트 갱신.

### 1-2. 속도 문제 → 추론(thinking) 모드 끔
가장 큰 병목이었다. 기본 모델 `qwen/qwen3.7-plus`는 **추론 전용 모델**이었다.

- **진단 (OpenRouter generation 통계)**:
  - `native_tokens_reasoning: 4017` — 답을 내기 전에 4천 토큰을 "생각"하는 데 소모
  - `generation_time: 95251ms` — **요청당 약 95초**
  - `finish_reason: "stop"` — 정상 종료지만 사고 텍스트가 `content`에 섞여 JSON 파싱을 방해
- **조치**: 백엔드 OpenRouter 요청에 `reasoning: { enabled: false }` 추가
  (`AiController.callOpenRouter`). 이 용도(계획 JSON 생성)엔 추론이 불필요하고, 켜두면
  수십 초 지연 + 사고 텍스트 혼입만 유발한다. 지원하지 않는 모델은 이 값을 무시한다.
- **결과**: 응답 속도가 실사용 가능한 수준으로 단축(브라우저 체감 "꽤 괜찮음").

> 교훈: **"똑똑한 모델 = 좋은 선택"이 아니다.** 정형 JSON을 빠르게 뽑는 작업엔
> 추론 모델이 오히려 독(지연·비정형 출력)이 된다. 작업 성격에 모델/모드를 맞춰야 한다.

### 1-3. 파라미터 확인 & 매칭 (응답 스키마 불일치)
화면이 하얗게 죽는(white-screen) 문제의 실제 원인. 데이터는 정상적으로 왔으나 **형태가 프론트 기대와 달랐다.**

- **기대한 형태**: `{ "tasks": { "2026-07-13": [ {id,content,completed}, ... ] } }` (날짜 → 배열 맵)
- **모델이 실제 준 형태**: `{ "plan": [ { "date": "2026-07-13", "tasks": [...] }, ... ] }` (배열)
- 프론트가 날짜 대신 `"plan"`을 키로 잡아 **"Day 1 · plan"** 만 뜨고 할 일은 안 나옴.
  또는 파싱 실패 시 `Object.entries(undefined)`로 앱 전체가 죽음.
- **조치**:
  - `coerceToDateMap` — `{plan|days|schedule|checklist|tasks|items:[...]}` 래퍼,
    `[{date,tasks},...]` 배열, `{tasks:{날짜:[...]}}`, 최상위 날짜맵을 **모두 날짜맵으로 흡수**.
  - `normalizeTasks` — 위를 거쳐 `{id,content,completed}`로 검증. 유효 항목이 없으면 `null` → mock 폴백.
  - 렌더링 방어: `Object.entries(tasks || {})`, `Array.isArray(list)` 체크.
  - 백엔드 `sanitizeJson` — 코드펜스 제거 후에도 순수 JSON이 아니면 **중괄호 균형으로
    최상위 `{...}`만 추출**(설명/사고 텍스트가 섞여도 JSON만 확보).
- 4가지 실제 응답 형태로 변환을 단위 검증하여 통과 확인.

### 1-4. 하루 할 일 개수를 투자 시간에 비례
고정 2~3개 → `dailyHours` 비례. 초안/수정/자유대화 프롬프트와 mock 폴백에 동일 기준 적용.

| 하루 투자 시간 | 하루 할 일 개수 |
| :--- | :--- |
| 1시간 이하 | 1~2개 |
| 2시간 | 2~3개 |
| 3~4시간 | 3~4개 |
| 5~6시간 | 4~5개 |
| 7시간+ | 5~6개 |

(구현: `AiController.tasksPerDayRange` / 프론트 mock 동일 기준. 사용자가 "늘려/줄여" 하면 그 요청 우선.)

### 1-5. 인프라 / 배포 안정화
- **CI/CD**: `main` 푸시마다 GitHub Actions가 이미지를 빌드해 `ghcr.io`에 push
  (`.github/workflows/image.yml`). VM은 빌드 없이 `deploy/oci-pull.sh`로 **pull만**.
  → 956MB VM이 gradle/npm 빌드로 마비되던 문제를 근본 해결.
- **키 관리**: `~/.delaynomore.env`(chmod 600) 자동 로드 → 매 배포 시 키 입력 불필요,
  셸 히스토리에 키 미노출. 빈 env를 컨테이너에 넘겨 Spring 기본값이 무시되던 버그 수정.
- **런타임**: `-XX:MaxRAMPercentage=50`으로 저사양 VM에서 JVM 힙 상한, 스왑 2G 추가.

## 2. 현재 AI 파라미터 상태 (무엇이 어디서 정해지는가)

| 파라미터 | 현재 값 / 방식 | 위치 | 제어 가능성 |
| :--- | :--- | :--- | :--- |
| `model` | `qwen/qwen3.7-plus` (기본) | `application.yml` `${OPENROUTER_MODEL:...}` | 가능 — 환경변수로 배포 시 교체 |
| `reasoning.enabled` | `false` (하드코딩) | `AiController.callOpenRouter` | 코드 수정 필요 |
| `temperature` | 미지정 (프로바이더 기본) | — | 없음 |
| `max_tokens` | 미지정 (프로바이더 기본) | — | 없음 (추론 모델에서 잘림 위험 있어 신중히) |
| 하루 할 일 개수 | 시간 비례 테이블 (하드코딩) | `tasksPerDayRange` | 코드 수정 필요 |
| 계획 기간/시간/수준 | 유저 입력 (슬롯필링) | 프론트 → 요청 바디 | 가능 — 런타임 |

즉 지금은 **모델만 환경변수로 외부화**되어 있고, 나머지 생성 파라미터는 코드에 박혀 있다.

## 3. 앞으로 파라미터를 어떻게 제어할 것인가 (방향 정리)

우선순위 순으로 단계적으로 도입하는 것을 제안한다.

### 3-1. (1순위) 생성 파라미터를 설정으로 외부화
모델처럼 `application.yml` + 환경변수로 빼서, **재배포만으로** 튜닝 가능하게 한다.

```yaml
openrouter:
  api:
    model: ${OPENROUTER_MODEL:qwen/qwen3.7-plus}
  params:
    temperature: ${OPENROUTER_TEMPERATURE:0.7}
    max-tokens:  ${OPENROUTER_MAX_TOKENS:}      # 비우면 미전송(프로바이더 기본)
    reasoning-enabled: ${OPENROUTER_REASONING:false}
```
- `AiController`가 이 값들을 읽어 요청 바디에 조건부로 실어 보낸다(빈 값은 생략).
- 장점: 모델 교체 시 함께 조정(예: 비추론 모델로 바꾸면 `reasoning` 무의미해도 안전).

### 3-2. (2순위) 하루 할 일 개수 테이블을 설정화
`tasksPerDayRange`의 매핑을 코드 상수가 아니라 설정(예: `plan.tasks-per-hour` 규칙)으로 빼서,
운영 중 "하루 개수 정책"을 코드 수정 없이 바꾼다. 또는 규칙을 단순 공식(`clamp(round(h*0.8),1,6)`)으로.

### 3-3. (3순위) 관측(Observability) 먼저, 튜닝은 데이터로
지금은 느린지/토큰을 얼마 쓰는지 로그가 없다. 튜닝의 근거를 남긴다.
- 백엔드에서 OpenRouter 응답의 `usage`(prompt/completion 토큰)와 소요 시간을 로그로 남긴다.
- 요청당 지연·토큰·모델을 남겨두면 "어느 모델·파라미터가 빠르고 싼지"를 데이터로 결정할 수 있다.
- (경험칙) 이번처럼 `generation_time`, `reasoning_tokens` 한 줄만 봐도 원인 규명이 즉시 됐다.

### 3-4. (4순위) 요청별(runtime) override — 신중히
프론트 "고급 옵션"에서 temperature·모델 등을 요청 바디로 넘겨 즉시 실험.
- 장점: 재배포 없이 A/B. 단점: **비용/보안 리스크**(임의 모델·과도한 max_tokens 요청 남용).
- 도입 시 **서버 화이트리스트/상한**(허용 모델 목록, max_tokens 캡)으로 가드해야 한다.
- 데모 단계에선 과함 → 3-1(설정 외부화)로 충분. 사용자 계정/설정이 생기는 단계에서 검토.

### 3-5. (참고) 모델 선택 가이드
- **정형 JSON을 빠르게** 뽑는 지금 용도 → **비추론 instruct 모델**이 정답에 가깝다.
- 추론 모델을 꼭 써야 한다면 `reasoning.enabled=false`가 먹히는지 반드시 확인
  (안 먹히면 여전히 느림 → 모델 교체).
- 무료(`:free`) 모델은 JSON 준수/한국어 품질이 들쭉날쭉 → 품질이 중요하면 소액 유료 모델 고려.

## 4. 한 줄 요약
느림의 원인은 **모델 성능이 아니라 "추론 모드"**였고(→ 껐다), 화면이 안 뜬 원인은
**출력이 아니라 "스키마 불일치"**였다(→ 정규화로 흡수). 앞으로는 이런 파라미터(모델·추론·
temperature·개수 정책)를 **코드가 아니라 설정으로** 빼서, 재배포만으로 튜닝하고
**로그로 근거를 남겨** 데이터로 결정한다.
