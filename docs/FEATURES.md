# 기능 상세

화면은 **가로 3칸**입니다 — **왼쪽=AI 코치와의 대화**, **가운데="오늘 할 일"**(보관된 계획들의 오늘 항목 모음), **오른쪽=생성된 체크리스트**. 모바일 폭에서는 같은 순서로 위아래 스택됩니다.

1. AI 코치가 4가지를 순서대로 질문합니다 — **목표 → 기간 → 하루 투자 시간 → 현재 수준**.
2. 입력이 모두 채워지면 백엔드가 OpenRouter로 하루 단위 계획 초안(투두리스트)을 생성합니다.
3. 초안 생성 후에는 **자유 대화 모드**가 됩니다 — LLM이 메시지의 의도를 판단해, 수정 요청이면 계획을 고치고 무엇을 바꿨는지 설명하고(예: "주말은 빼줘"), 질문이면 답하고, 불명확하면 되묻습니다.
4. `OPENROUTER_API_KEY`가 없거나 백엔드가 응답하지 않으면 프론트가 **템플릿 기반 mock 계획**으로 자동 폴백하여 데모 흐름이 끊기지 않습니다.
5. **여러 계획 보관(브라우저별 보관함)** — 완성된 계획은 서버 보관함에 **자동 보관**되고, 이후의 수정·완료 체크·고정도 자동 동기화됩니다. 우측 패널의 "보관된 계획" 목록에서 여러 계획을 전환/삭제할 수 있고(소유자당 최대 10개), 새로고침하면 마지막으로 보던 계획이 복원됩니다. 보관함은 브라우저 게스트 ID별로 격리되며(아래 11번), 저장소는 배포 프로필에 따라 다릅니다 — 로컬/단위 테스트 기본값은 **서버 메모리에만 휘발성으로** 보관해 서버 재시작 시 초기화되고, 배포 환경(`postgres` 프로필)은 PostgreSQL에 영속되어 서버 재시작에도 유지됩니다(v0.12.0, 로그인은 계속 미도입).
6. **"계획 저장" 버튼은 계획을 고정(확정)합니다** — 고정 후에는 대화로 계획을 수정할 수 없고(강제성 부여: 확정한 계획은 재협상 없이 실행), 완료 체크와 질문만 가능합니다. 바꾸려면 "처음부터 다시 만들기"로 새 계획을 시작하세요(이전 계획은 보관함에 남습니다).
7. **오늘 할 일(오늘 보기)** — 화면 가운데 칸이 **모든 보관된 계획**에서 오늘 날짜의 할 일만 모아 계획 이름과 함께 보여줍니다. 밴드에서 바로 완료 체크할 수 있고(원본 계획과 즉시 양방향 반영 + 서버 저장), "오늘 n/m 완료" 진행률과 "계획 보기" 이동 버튼을 제공합니다. 오늘 항목이 없으면 "오늘 할 일이 없습니다"로 안내합니다.
8. **"오늘 마무리" 일일 회고** — 오늘 할 일 칸 하단에서 하루를 닫으며 회고를 남깁니다. 완료 결과("오늘 N개 중 M개 완료 · 완료율 %")는 자동 계산되고, 체감 난이도(3택1)와 이유(5택1)만 선택해 저장합니다(자유 입력 메모 없음). 회고는 계획별·날짜별 1건만 유지(재저장 시 갱신)되고, 저장은 **오늘(한국 시간 기준) 날짜만** 허용되며, 완료 개수는 서버가 계획에서 재계산합니다. 각 계획 카드의 **"지난 회고"**를 펼치면 그 계획의 과거 회고를 최신순으로 되짚어 볼 수 있습니다. 회고는 계획의 소유자 격리를 그대로 따르며(내 계획의 회고만 보임), 계획과 같은 저장소를 쓰므로 저장 지속성도 위 5번과 동일합니다(배포 환경은 PostgreSQL 영속, 로컬/단위 테스트는 휘발성).
9. **미완료 항목 내일로 이동** — 오늘 할 일 칸의 계획 그룹에 미완료가 있으면 "미완료 N건 내일로" 버튼이 나타납니다. 확인 후 오늘의 미완료 항목만 내일로 이월되고(완료 항목은 오늘에 잔류), 내일이 계획 기간을 넘으면 종료일·기간이 하루 연장됩니다. **고정(CONFIRMED)된 계획은 이월할 수 없습니다** — 계획 구조를 바꾸는 행동이라 버튼이 표시되지 않습니다(고정 시 수정 버튼을 숨기는 관례와 동일).
10. **계획 변경 이력** — 보관된 계획 행의 이력 버튼(🕘)으로 그 계획의 변경 기록을 펼쳐 봅니다: 생성·수정·고정·할 일 완료/해제·회고 저장·삭제가 각각 언제, "이 브라우저"에서인지 "다른 세션"에서인지 표시됩니다(같은 게스트 ID를 여러 탭에서 쓸 수 있으므로). 이벤트는 서버가 변경 시점에 직접 기록하며(조회 등 읽기 행동과 AI 대화 전문은 기록하지 않음), 이력 조회도 소유자 격리를 따르되 이벤트에 소유자가 기록되어 **삭제된 계획의 이력도 소유자에게는 조회됩니다**(저장 지속성은 위 5번과 동일 — 로컬/단위 테스트 기본값은 전체 최근 1,000건 상한 + 서버 재시작 시 초기화되는 휘발성 저장소, 배포 환경(`postgres` 프로필)은 상한 없이 PostgreSQL에 영속).
11. **브라우저별 개인 보관함(닉네임 표시)** — 최초 진입 시 닉네임(한글·영문·숫자 2~20자)을 설정합니다. 데이터 소유·격리의 기준은 브라우저가 생성한 **게스트 ID**(X-Guest-Id 헤더)이고, 닉네임은 화면에 보이는 표시 이름일 뿐 서버로 가지 않습니다. 따라서 계획·회고·이력이 브라우저별로 격리되어 **다른 브라우저에서는 같은 닉네임을 써도 별도의 보관함**이 되고, 헤더의 "변경"으로 닉네임(표시 이름)만 바꿔도 데이터는 그대로 유지됩니다. 이는 **로그인 전 브라우저 단위 임시 개인 보관함**입니다 — v0.12.0부터 서버 쪽 데이터는 PostgreSQL에 영속되어 서버 재시작으로는 사라지지 않지만, **브라우저에 저장된 게스트 ID 자체를 잃으면**(브라우저 데이터 삭제, 다른 브라우저·기기로 접속, 프라이빗 모드 종료) 닉네임을 기억하고 있어도 자동으로 재연결할 방법이 없습니다(닉네임은 서버로 전송되지 않아 소유자 조회 키가 될 수 없음 — HTTP 데모라 민감한 정보 저장 금지). 로그인 도입 시 게스트 ID의 데이터가 계정(memberId)으로 이전됩니다.
12. **다음 계획 분량 추천(수행 기록 기반)** — 완료했거나 3일 이상 실행한 계획 행에 **✨ 추천 버튼**이 뜹니다. 누르면 서버가 지난 계획의 수행 기록(하루 평균 개수·완료율·"벅찼어요" 일수·주요 회고 이유)을 계산하고, **규칙으로 다음 하루 분량을 결정**합니다. 표본이 작아 완료율이 튀지 않도록 **같은 목표(goalName)의 최근 계획을 최대 3건까지 합산**해 완료율·관찰일수·회고를 집계합니다(v0.13.1 — 하루 평균은 가장 최근 계획 기준, 계획이 1건이면 단일 계획과 동일. 목표명을 바꾸면 합산 그룹이 갈라짐) — 완료율이 낮거나 "분량 많음" 회고가 반복되면 1개 줄이고, 완료율이 높고 여유로웠다는 회고가 많으면 1개 늘립니다(하루 1~5개, 한 번에 ±1). **분량 숫자는 서버 규칙이 정하고 AI는 이유 설명과 내용 생성만** 합니다(AI가 없어도 규칙 결과와 기본 설명이 뜨고, 초안도 서버 템플릿으로 채워집니다). `[추천대로]·[기존처럼]·[직접 설정]` 중 고르면 그 분량대로 초안이 생성되고, **미리보기에서 "이 계획 저장"을 눌러 승인해야만** 새 계획으로 저장됩니다(승인 전에는 저장되지 않고, 원본 계획·회고는 바뀌지 않습니다). 조회·선택·생성은 변경 이력에 남습니다.
