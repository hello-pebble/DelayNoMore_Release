# QA 테스트 — v0.13.0 다음 계획 분량 추천

- **대상 버전**: `v0.13.0`
- **범위**: AI 다음 계획 분량 추천(로드맵 4·5번) — 완료했거나 3일 이상 실행한 계획에서 서버가
  수행 기록을 계산하고 규칙으로 다음 하루 분량을 결정, AI는 이유·내용만, 승인 전 미저장.
- **사용법**: 배포 URL(`http://<배포주소>`)에 접속해 순서대로 확인. 대부분 브라우저에서 되고,
  서버 전용 조건 일부는 `curl`을 쓴다.
- **관련 문서**: 전체 QA 체크리스트는 [QA_CHECKLIST.md](QA_CHECKLIST.md) F-22, API는
  [API_REFERENCE.md](API_REFERENCE.md) "다음 계획 분량 추천" 참고.

> **curl 주의**: 계획 API는 필수 헤더 `-H "X-Guest-Id: <UUID>"`가 필요하다. 내 guestId는 브라우저
> 콘솔에서 `localStorage.getItem('delaynomore:guestId')`로 확인한다.

## 준비 — 추천 버튼을 띄우는 가장 빠른 길

"3일 이상 실행"은 날짜가 지나야 하지만, **"완료(고정 + 전부 완료)"** 조건이면 바로 뜬다.

- [ ] 새 계획을 하나 만든다(예: 기간 2일).
- [ ] 오른쪽 체크리스트에서 **모든 할 일을 완료 체크**한다.
- [ ] **"계획 저장"(고정)** 을 누른다 → 보관함 그 계획 행에 **✨ 버튼**이 나타난다.

## A. 버튼 노출 (recommendationEligible)

- [ ] 위에서 만든 **완료·고정 계획** 행에 ✨ 버튼이 보인다.
- [ ] 방금 만든 **짧은 미완료 초안**(고정 안 함, 오늘 하루짜리) 행에는 ✨ 버튼이 **안 보인다**.

## B. 추천 조회 → 통계·이유 (완료 조건 6 일부)

- [ ] ✨ 버튼 클릭 → **"지난 계획"** 통계(하루 평균 N개 · 완료 X/Y · %)와 **추천 이유** 문장,
      그리고 `[추천대로]·[기존처럼]·[직접 설정 1~5]` 선택지가 뜬다.
- [ ] 이유 아래에 **"AI 설명" 또는 "규칙 기반 설명" · 분량은 서버 규칙이 결정합니다** 표기가 보인다.

## C. 선택 → 미리보기 → 승인 저장 (완료 조건 7·8·9)

- [ ] `[직접 설정]`에서 **2**를 누른다 → "초안을 생성하는 중..." 후 **미리보기**가 뜬다.
- [ ] 미리보기의 **각 Day 항목 수가 정확히 2개**다(선택값과 일치).
- [ ] **이 시점에 오른쪽 보관함 목록을 보면 새 계획이 아직 없다**(승인 전 미저장).
- [ ] **"이 계획 저장"** 클릭 → 새 계획이 화면에 열리고 보관함 목록에 추가된다.

## D. 원본 불변 · 중복 방지 (완료 조건 10·11)

- [ ] 저장 후 **원본(추천을 받았던) 계획**을 열어보면 내용·완료 상태가 **그대로**다.
- [ ] (중복) 미리보기에서 "이 계획 저장"을 **빠르게 두세 번 클릭** → 버튼이 "저장 중"으로 잠기고
      **새 계획이 하나만** 생긴다.

## E. 변경 이력 (완료 조건 12)

- [ ] 저장된 **새 계획**의 🕘 이력 → **"추천 분량 채택"**(또는 직접 설정으로 추천과 다르게
      골랐으면 "추천 분량 변경") + **"추천 기반 계획 생성"** 이 보인다.
- [ ] 추천을 받았던 **원본 계획**의 🕘 이력 → **"다음 분량 추천 조회"** 가 보인다.

## F. AI 실패 폴백 (완료 조건 6)

`OPENROUTER_API_KEY`가 없거나 AI가 다운된 상태에서도 동작해야 한다.

- [ ] (헤더의 AI 상태 LED가 빨강/미연결이거나 키 미설정 배포에서) ✨ 추천 → 통계 +
      **규칙 기반 설명**이 뜬다.
- [ ] 분량 선택 → **"기본 템플릿"** 표기가 붙은 초안이 생성되고, 승인 저장까지 정상 동작한다
      (패널이 죽지 않는다).

## G. 소유자 격리 (완료 조건 1) — curl

- [ ] 내 `guestId`가 아닌 **다른 UUID**로 남의 계획에 추천 요청 → **404 `PLAN_NOT_FOUND`**:

```bash
curl -s -X POST "http://<배포주소>/api/v1/plans/<계획id>/recommendation" \
  -H "X-Guest-Id: 00000000-0000-0000-0000-000000000000" | jq
# → { "success": false, "error": { "code": "PLAN_NOT_FOUND" } }
```

## 참고 — 자동 테스트로 이미 커버된 규칙 분기 (수기 QA 선택)

아래는 날짜/회고 데이터를 3일 이상 쌓아야 재현되므로 수기 QA가 번거롭다. 규칙 로직
(완료 조건 2~5)은 백엔드 단위 테스트 `WorkloadRecommendationTest`(9건)로 전 분기를 검증했으니
**필수는 아니다**. 실데이터로 확인하려면 시작일이 3일 전인 계획에 회고를 며칠 저장한 뒤 추천을
받아 `completionRate`와 `recommendedTasksPerDay`를 비교한다.

| # | 규칙 | 기대 결과 |
|---|---|---|
| 2 | 관찰 3일 미만 | `insufficientHistory: true`, 추천 = 기존 분량(변화 없음) |
| 3 | 미래 날짜 항목 존재 | 완료율·관찰일수에서 제외(오늘 이하만 집계) |
| 4 | 완료율 50% 미만 | 하루 1개 감소(최소 1) |
| 5 | 완료율 85%+ & EASY 회고 절반 이상 | 하루 1개 증가(최대 5) |

> 참고: 완료율 50~85%(중립)면 유지. 벅찬 회고 절반 이상 + "분량 많음" 2회 이상이면 중립·증가
> 판정을 덮어쓰고 감소. 한 번에 최대 ±1개, 하루 1~5개로 클램프.
